# ストリート野球

## 開発環境
- Godot 4.6

## ゲーム概要
`ストリート野球` は、ヤンキーや一般人による独自野球リーグを舞台にした球団運営シミュレーションです。
ルールや年間進行は NPB 準拠ベースを土台にしつつ、ところどころにストリート感や遊び心のあるイベントを入れていく方針です。

プレイヤーは担当球団を選び、その球団を中心に以下を進めます。
- 日々の進行と試合消化
- 編成管理
- ライブ試合での采配
- 球団運営
- 複数年プレイ

## 現在の大きな方針
- まずはゲーム全体の型を完成させる
- 細かい微調整や数値バランスは後回しにする
- ライブ試合は本命、過去試合は情報確認に寄せる
- 365日進行のカレンダーを土台にする
- ホームはパワプロ寄り、管理画面は OOTP 寄りで設計する

## 再設計方針
### 方針
- 全面作り直しはしない
- 画面構成と年間カレンダーの資産は残す
- 中核ロジックだけを段階的に整理し直す
- まずは `1年を回すゲーム` として破綻しないことを優先する

### 残すもの
- 画面構成の考え方
  - ホームはパワプロ寄り
  - 管理画面は OOTP 寄り
- 年間カレンダーと年間イベントの大枠
- 12球団 / 2リーグ制
- 固定開幕ロスターの方針
- ロスター管理、契約・FA、スカウト・ドラフト、記録室などのページ分離

### 整理し直すもの
- `LeagueState` に集中している中核ロジック
- 契約更改 / FA / ドラフト / ロスター管理の相互依存
- オフ処理と年越し処理
- 画面から直接データをいじっている箇所の一部

### これからの分割イメージ
- `RosterManager`
  - 一軍 / 二軍 / 育成 / 支配下 / 外国人枠
- `ContractManager`
  - 契約更改 / FA / 年俸 / 契約満了
- `DraftManager`
  - 候補 / 注目候補 / 指名 / 指名結果反映
- `CalendarManager`
  - 年間イベント / 節目 / 期間判定
- `HistoryManager`
  - シーズン履歴 / 運営ログ / 球団史

### 実際の進め方
1. まず今の画面は維持したまま backend の責務を整理する
2. `LeagueState` からロスター / 契約 / ドラフト / カレンダー処理を順に切り出す
3. 切り出した backend に各画面をつなぎ直す
4. その後で細かい能力差や試合影響を厚くする

### 優先順位
1. 既存選手の運用
  - 一軍 / 二軍 / 育成
  - スタメン
  - 先発6枚
  - 勝ちパターン
2. 契約更改 / FA / ドラフト
  - 年に一回回る仕事として成立させる
3. 育成
  - 施設 / スタッフ / 育成方針
4. 記録 / 球団史 / 年間ログ
5. 試合と能力の細かい接続

## コアループ
1. 担当球団を選ぶ
2. ホーム画面で日付と今日の予定を確認する
3. 必要ならチーム管理・球団運営を行う
4. 試合を見るか、自動進行で進める
5. シーズンを越えて複数年運営する

## 年間サイクル
### 月ごとの大きな流れ
- 1月: スタッフ見直し、オフの整理、来季準備
- 2月: 春季キャンプ
- 3月: オープン戦、開幕準備
- 4月〜5月前半: レギュラーシーズン前半
- 5月下旬〜6月中旬: 交流戦
- 6月中旬〜10月上旬: レギュラーシーズン後半
- 10月中旬: クライマックスシリーズ
- 10月下旬〜11月上旬: 日本シリーズ
- 10月下旬: ドラフト準備、ドラフト会議
- 11月: 契約更改、FA交渉
- 12月: スポンサー更改、オフの整理

  - 指名方針の確認

  - 指名実行

- 契約更改
  - 残留交渉

  - FA流出対策

  - 収益改善

## 画面構成
### TeamSelect
- 担当球団選択画面

### LeagueHome
- ホーム画面
- 日付、担当球団、進行ボタン、今日の予定、昨日の結果、ニュース、簡易速報を表示
- カレンダー感を重視し、今の時期に触るべき運営項目への導線を出す
- `今日の試合へ` からライブ試合へ入る

### TeamManagement
- 編成管理画面
- 打線、ローテ、ブルペン、方針を管理

### LeagueInfo
- リーグ全体の情報画面
- 順位、個人成績、年間日程、ニュース、試合詳細を確認
- `概況 / 日程 / 個人成績 / 履歴` タブで切り替える

### CalendarScene
- 年間カレンダー画面
- 今日の年間イベント
- 次に来るイベント
- 月ごとに切り替えて、その月のイベントと担当球団予定を確認

### RosterView / PlayerDetail
- 選手一覧と選手詳細
- 年齢、年俸、能力、起用状況、成績などを表示
- `選手一覧` は見る場所
- `選手詳細` は `基本 / 能力 / 特殊能力 / 成績` タブで切り替える

### RosterManagement
- ロスター管理画面
- 一軍 / 二軍 / 育成 / 支配下 / 外国人枠を管理
- ここで一軍登録、二軍移動、支配下登録、育成方針変更を行う

### GameDetail
- 過去試合の詳細確認画面
- スコア、勝敗投手、詳細ログを表示

### MatchScene
- 当日の担当球団ライブ試合画面
- 一球ごと / 一打席ごと進行
- オート速度切替
- 停止場面設定
- 采配・指示
- 試合後サマリー

### 球団運営系画面
- FrontOffice: ハブ画面
  - 年間イベントに応じて、今が使いどきの項目を強調表示
  - ContractOffice: 契約更改と FA 導線
  - ContractRenewalOffice: 契約更改の実行ページ
  - FAMarket: FA 市場の実行ページ
  - TradeOffice: 簡易トレードの実行ページ
  - FacilityOffice: 施設強化
  - SponsorOffice: スポンサー営業
  - StaffOffice: スタッフ調整
  - ScoutDraftOffice: スカウト / ドラフトのトップ画面
- RecordRoom: 今季、歴代、球団史、運営ログを確認するページ

## 試合実装の考え方
### 基本方針
- 試合ロジックと演出を分ける
- まず内部で試合イベント列を作る
- そのイベントを画面で順に再生する
- 最初は試合の型を完成させ、後から演出を足す

### イベントの粒度
- まずは打席単位
- 次に一球ごとの `pitch` イベントを入れる
- 最終的にはライブ采配で結果分布を変える

### 指示の影響先
試合中の指示は `能力 + 状況 + 指示` を合わせて結果分布を動かす前提です。
主な影響先は以下です。
- 打席結果の確率
- 進塁処理
- 球数や一球イベント
- 起用や試合運び

例:
- `強攻`: 長打と三振が増えやすい
- `バント`: バント専用結果に寄せる
- `進塁重視`: 進塁打や単打寄りにする
- `敬遠`: 四球寄りにする

### 試合画面の役割分離
- 過去試合は `GameDetail` で確認する
- ライブ試合は `MatchScene` に集約する
- `MatchScene` は一発勝負の前提で、巻き戻しや再入場で結果を変えられないようにする

## NPB準拠ベースの初期12球団方針
### 参照方針
- NPB公式のチーム構成と公示を参考にする
- 実在チーム名やロゴはそのまま使わず、地域性や球団カラーだけを借りた架空球団にする
- 現役選手の完全再現ではなく、チームカラーに寄せた架空ロスターを組む

### 初期12球団の方向性
- 2リーグ制、各6球団
- 東京圏2球団
- 大都市圏と地方人気球団を混ぜる
- 資金力、人気、育成、投手力、機動力などに球団カラー差を持たせる

### ロスター設計ルール
- 支配下相当: 68人前後を基本
- 上限想定: 70人
- 一軍相当: 29人前後
- 一軍内訳の目安
  - 投手 14人
  - 野手 15人
- ローテ: 6人
- 主要ブルペン: 8人前後
- ベンチ: 7人前後

### 外国人枠の方針
- 保有人数: 4人前後から開始
- 一軍同時登録: 最大4人
- 投手最大3
- 野手最大3

### 実装順
1. 12球団の青写真を用意する
2. 開幕時の固定ロスターを用意する
3. 外国人枠と登録人数ルールを入れる
4. その後で 6球団仮リーグから 12球団リーグへ移行する

### 固定とランダムの使い分け
- 開幕時の球団と主力ロスターは固定にする
- 球団の個性や戦力差は、最初から毎回同じ状態で始まるようにする
- ランダム要素は後から入れる
  - スカウト候補
  - ドラフト候補
  - 外国人助っ人
  - FA市場
  - トレード候補

## 画面構成方針
### レイアウト思想
- `ホーム` はパワプロのペナントや栄冠ナインのように、日付と今日の予定を中心に進める
- `球団運営` は OOTP の Team Menu のように、管理機能へのハブ画面として使う
- `選手一覧` は見る場所、`ロスター管理` は動かす場所、というように役割を分ける
- できるだけ「今どこで、何をする画面か」が1画面で分かる構成を優先する

### 画面の見せ方ルール
- `判断するページ` は、できるだけ 1 画面で全体像が分かるようにする
- `情報量が多いページ` は、上部タブで切り替える
- `掘るページ` は、縦スクロール前提でよい
- 基本思想は `ホームはパワプロ寄り`, `管理画面は OOTP 寄り`
