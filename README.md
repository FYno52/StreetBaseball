# ストリート野球

## 開発環境

- Godot 4.6

## コンセプト

『ストリート野球』は、ヤンキーや一般人たちが参加する独自の野球リーグを舞台にした野球シミュレーションゲームです。

リーグの基本構造、試合進行、成績管理、シーズン運営の考え方は NPB をベースにします。
一方で、世界観や演出、イベント、特殊ルールの一部にはストリート野球らしい遊び心やハプニング要素を加え、まじめな野球シミュレーションの中に少し外した面白さを混ぜていく方針です。

現段階では、まずリーグ戦と試合シミュレーションの基盤を安定させることを優先し、特殊要素は後から段階的に追加していきます。

## ゲーム概要

Godot で開発中の野球シミュレーションゲームです。
現在はリーグ戦を自動生成し、各チームの戦力から試合結果を算出して、順位表や個人成績を更新する土台が実装されています。

このプロジェクトは、まず軽量なシミュレーションでシーズン進行を成立させ、そのあとに起用、育成、戦術、演出、ストリート野球らしい特殊要素を段階的に追加していく方針です。
最終的には、プレイヤーが担当する球団を1つ選び、その球団を中心にシーズンを進めていく球団経営シミュレーションとして仕上げる想定です。
また、シーズン進行は将来的に `2026年開始` の日付ベース進行へ寄せ、数十年単位で回せる長期運営型にしていく方針です。

## 世界観メモ

- 参加者はプロ野球選手ではなく、ヤンキーや一般人を中心とした独自リーグの選手たち
- リーグ運営や成績管理は NPB 的な整理された形を参考にする
- 試合の基本ルールは野球らしさを保つ
- ただし世界観やイベントでは、ストリートらしい荒さ、個性、意外性を入れる
- 商店街、河川敷、地元客、先輩後輩、差し入れ、口論など、街の空気が試合に少し乗る作りを目指す
- 将来的には特殊イベント、ローカルルール、クセの強い選手特性なども検討する

## 現在の実装範囲

- チーム、選手、試合データの基本クラス
- ランダムなチームロスター生成
- 総当たり日程の自動生成
- 先発投手と打線能力を使った簡易試合シミュレーション
- 左右別打線の基礎対応
- 勝敗、得失点、最低限の打撃/投手成績更新
- リーグ順位表示
- チーム詳細、左右別打線、ローテーション表示
- 直近試合結果の表示
- セーブ/ロード用のデータ変換

## コアループ

想定している基本の遊びは次の流れです。

1. 担当球団を選ぶ
2. 担当球団の状況とリーグ全体の流れを確認する
3. 打線、ローテ、ブルペン、方針を調整する
4. 試合を1日進める
5. 担当球団の戦績、選手成績、資金、人気の変化を見る
6. シーズン終了まで進行する

将来的にはこのループに、編成、育成、補強、戦術、特殊イベントの判断を加えていきます。

## 暫定完成までのステップ

細かい演出やイベント量より先に、まず `一旦完成` と呼べる型を作るため、当面は次の順で進めます。

1. 年間365日進行と NPB 準拠ベース日程を安定させる
2. 担当球団を中心に、ホーム / チーム管理 / リーグ情報 / 選手一覧 の導線を固める
3. 1シーズン完走、年越し、次年度開幕まで止まらず回るようにする
4. 前年度の結果や担当球団の歩みが残る `シーズン履歴` を持たせる
5. そのあとで、イベント拡充、数値バランス、演出、細かいUI調整を進める

当面のゴールは、`担当球団を選んで数年進めても破綻しない、遊べる球団運営シム` を先に作ることです。
そのため、まずは `LeagueInfo` 上で複数年の結果を見返せる最低限の年度推移を残し、担当球団の順位・勝敗・人気・予算の変化に加えて、通算成績や優勝回数も追えるようにして、長期プレイの土台を優先して整えていきます。

## ゲーム進行の軸

このゲームは最終的に、リーグ全体を俯瞰するだけのシミュレーターではなく、`担当球団を持って進める` 形式を基本にします。

### 目指す形

- ゲーム開始時に担当球団を選ぶ
- ホーム画面やニュース、進行導線は担当球団を中心に表示する
- チーム管理画面では担当球団の編成と方針を調整する
- リーグ情報画面では順位表や個人成績、他球団の動きも確認できる
- シーズン進行、イベント、資金、人気の変化は担当球団に対する影響として体験する

### 設計への反映方針

- `LeagueState` に担当球団IDを持たせる前提で拡張する
- `LeagueHome` は担当球団中心のダッシュボードへ寄せる
- `TeamManagement` は担当球団の管理画面として整理する
- `LeagueInfo` はリーグ全体の確認画面として分離する
- 将来的なイベントや経営要素も、まず担当球団への影響を基準に設計する
- 年度は `2026` 開始を基準にし、日程データは日付ベースへ段階的に拡張する

現状では、担当球団IDを `LeagueState` で保持する方向に着手しており、ホーム画面の注目チーム表示もこの担当球団を基準に寄せていく予定です。
また、チーム管理画面から担当球団を設定できる導線を段階的に整えていきます。
担当球団が未設定の状態で起動した場合は、まず球団選択画面へ入る流れを基本にしていきます。
`LeagueInfo` は担当球団を直接操作する場所ではなく、順位表、個人成績、ニュース、試合結果を横断して確認するための画面として整えていきます。
実装順としては、まず `LeagueInfo` の見やすい骨組みとセクション構成を整え、その後にランキングや比較表示の中身を段階的に充実させていきます。

## カレンダー進行メモ

今後の日程システムは、いきなり完全な NPB 準拠へ振り切るのではなく、次の順で段階的に進めます。

1. `2026年開始` に揃える
2. 試合データに `日付` 情報を持たせる
3. 1日進行のUIでも `何月何日か` が分かるようにする
4. 休養日を含む年間カレンダーへ広げる
5. その後で NPB に近い連戦・移動日・長期年数プレイへ発展させる

現状はこのうち、`2026年開始` と `日付情報を持てる土台作り` を優先する段階です。
あわせて、`LeagueInfo` では担当球団の年間日程を確認できる表示土台を先に整え、後から月別やカード別の見やすさを上げていく方針です。
現段階では、まず月ごとにまとまりが見える一覧表示を優先し、その後に休養日表示や詳細導線を足していきます。
UI 表示は `何日目` を内部管理用に残しつつ、プレイヤー向けには `2026-03-27(金)` のような日付表記を主役にしていく方針です。
現時点の複数日進行では、`試合がある日` をまとめて進めつつ、結果表示では `経過した実日数` も併記して、365日ベース進行へ段階的に寄せていきます。
試合がない日も `日付としては進む` ことを前提にし、休養日や移動日を含む年間進行の型を先に固めます。
最終的には `シーズン終了ボタン` や `新シーズン開始ボタン` に頼らず、日付進行の中で開幕日になれば自然にシーズンへ入り、年末を越えれば次年度へ移る形を基本にします。
公式戦の日程は、まず `NPB準拠ベース` として年間 140 試合前後の感覚に寄せ、後から細かい並びやイベント日程を調整していきます。
日付進行で年末を越えた場合は、そのまま自動で次年度へ移行する形を基本にし、プレイヤーが季節の切り替えボタンを押さなくても年間ループが続く構成を目指します。
また、年間日程は単なる一覧ではなく、後から試合詳細へ自然につながる導線を持たせる前提で整えていきます。
`LeagueInfo` も日付ベースの年間進行に合わせて、開幕前 / シーズン中 / オフシーズン を自然に読み取れる構成へ段階的に揃えていきます。
オフシーズンはまず最低限の骨格として、年またぎ時の年齢更新、プロ年数更新、コンディション回復、人気・予算の年次精算を行う方針です。
その次の段階として、最低限の引退処理と新人補充を入れ、人数不足でシーズンが止まらない長期運営の土台を優先して整えていきます。
担当球団の経営表示は、まず試合ごとの簡易収支と年またぎ時の年次精算が分かる形から始め、後から詳細な経営システムへ拡張していきます。
オフシーズンには本格的な契約更改画面の前段階として、成績と能力に応じた `簡易年俸更改` を走らせ、さらに `予算に対して年俸総額が膨らみすぎない` よう自動調整を入れて、複数年運営で経営が破綻しにくい形を優先します。
担当球団のオフシーズンでは、最低限の引退・加入の動きがプレイヤーに見える形を優先し、後からドラフトや契約更改の画面へ発展させていきます。
選手一覧は、まず年齢・年俸・総合・役割が分かる形から始め、後から並び替えや能力比較、契約導線を強めていく方針です。
ゲームを回せることを最優先するため、ホーム画面には `1日進める` に加えて `7日進める` `次の試合まで` `次の節目まで` のような時短導線を段階的に入れていきます。
`次の試合まで` は意味がぶれないように `シーズン中のみ` 使える進行とし、別途 `オート進行` で 1 日ずつ表示を更新しながら進められる形も用意していきます。
`次の試合まで` は意味を明確にするため `シーズン中のみ` 使える進行とし、開幕前やオフシーズンは `次の節目まで` や `オート進行` で日付を進める前提にします。
また、年またぎやロード後に起用データが壊れて進行不能にならないよう、チーム構成を自動で整える安全策を優先して入れていきます。
同様に、年またぎやロード後でも試合データの日付表示が崩れないよう、`GameData` の日付メタデータは自動で再計算して揃える方針にします。
進行中に自動整形が走った場合は、ホーム画面のステータス欄に簡単な整形メモを出し、完走テスト時に異常の気配を追いやすくしていきます。
あわせて、進行後には簡易監査を走らせ、ローテや打線の不足など致命傷になりやすい状態をホーム画面で確認できるようにしていきます。

## 試合シミュレーション方針

現段階では、1打席ずつ処理する詳細シミュレーションではなく、チーム戦力ベースの軽量シミュレーションを採用しています。

### 現在の考え方

- 打線の攻撃力を平均化してチーム打力を作る
- 先発投手の能力から投手側の守備力を作る
- 左右別打線を相手先発の利き腕に応じて使い分ける
- 両者の差と乱数から得点を決める
- 試合後に最低限のチーム成績、個人成績を更新する

### 今後の拡張候補

- リリーフ、セットアッパー、クローザーの投入
- 疲労、コンディション、連投ペナルティ
- 守備力、走塁力、球場補正の反映
- 試合ログの詳細化
- 打席単位またはイニング単位への段階的拡張
- ストリート野球らしい特殊イベントやローカルルールの追加

### 試合実装の進め方メモ

試合部分は、いきなり映像つきの完成形を目指すのではなく、`試合ロジック` と `見せ方` を分けて段階的に作る方針です。

1. まず内部で試合結果を決める
2. 次に、その結果を `試合イベント列` として取り出せるようにする
3. 最後に、そのイベント列を UI や演出で順に再生する

#### 基本の考え方

- 映像が試合結果を決めるのではなく、`内部で決まった結果に演出を付ける`
- まずは `再生できる試合イベント` を作ることを優先する
- 最初はテキストとスコア表示だけでも十分に試合画面として成立させる

#### 段階的な実装順

1. `簡易打席単位` または `イニング単位` の試合進行を作る
2. 打席結果を決める処理を作る
   - 三振
   - 四球
   - 単打
   - 二塁打
   - 三塁打
   - 本塁打
   - 凡打
3. 走者進塁と得点処理を作る
4. 3アウトで攻守交代するイニング進行を作る
5. 9回までの試合進行を作る
6. 継投処理をつなぐ
7. `試合イベント列` を UI 側で再生する
8. 最後にアニメや演出を足す

#### 試合イベント列の役割

将来的な試合画面では、1試合ごとに次のようなイベントを順に再生する想定です。

- 試合開始
- 各イニングの進行
- 得点イベント
- 継投イベント
- 試合終了

このイベント列を元にして、

- スコアボード更新
- 塁状況表示
- テキスト実況
- 簡易アニメ
- 将来的な映像演出

を載せていきます。

また、最終的にはこの `試合イベント列` に対して、単に結果を再生するだけでなく、

- 采配
- 作戦指示
- 継投判断
- 代打 / 代走 / 守備固め
- ストリート野球らしい特殊判断

のようなプレイヤー操作や AI 判断を差し込み、その内容に応じて

- 打席結果の分岐
- 走者進塁の変化
- 継投タイミングの変化
- 表示される演出や実況の変化

が起こる形へ発展させていく方針です。

つまり最終的な試合画面は、`決まった結果にアニメを付けるだけ` ではなく、`采配や指示がイベント列そのものを変化させ、その結果として演出も変わる` 構造を目指します。

#### 最初の試合画面で必要な最小要素

- イニング
- アウト
- 塁状況
- スコア
- 打者 / 投手
- 打席結果ログ
- `次へ` ボタン
- `オート` ボタン

この段階では、フルアニメーションよりも `試合が正しく進み、読めること` を優先します。
現状はその前段階として、`LeagueInfo` で試合一覧を見て、`試合詳細` でスコアや投手結果を確認し、その先の `試合再生` 画面で保存された `play_events` を順に確認できる最小構成を先に用意していきます。

## データ設計メモ

### TeamData

チームが持つ主な情報です。

- 基本情報: `id`, `name`, `short_name`
- 運営情報: `budget`, `fan_support`, `strategy`
- ロスター情報: `player_ids`
- 起用情報: `lineup_vs_r`, `lineup_vs_l`, `bench_ids`, `rotation_ids`, `bullpen`
- 成績情報: `standings`

### PlayerData

選手が持つ主な情報です。

- 基本情報: `id`, `full_name`, `age`, `bats`, `throws`
- 役割情報: `role`, `primary_position`, `secondary_positions`
- 成長情報: `overall`, `potential`, `development_type`
- 状態情報: `morale`, `fatigue`, `injury_risk`, `condition`
- 契約情報: `salary`, `years_pro`
- 能力値: 打撃、守備、投球関連の `ratings`
- 補助情報: `pitch_types`, `traits`
- 成績情報: `batting_stats`, `pitching_stats`

### GameData

試合が持つ主な情報です。

- 対戦情報: `day`, `away_team_id`, `home_team_id`
- 結果情報: `played`, `away_score`, `home_score`
- 投手記録: `winning_pitcher_id`, `losing_pitcher_id`, `save_pitcher_id`
- 表示用ログ: `log_lines`

## システム構成

- `LeagueState`
  - リーグ全体の進行、チーム/選手/日程の管理、セーブ/ロードを担当
- `SimulationEngine`
  - 1試合の結果計算と成績反映を担当
- `PlayerFactory`
  - 新規リーグ開始時のロスター自動生成を担当
- `TeamData`, `PlayerData`, `GameData`
  - シミュレーションで扱うデータ構造

## 画面設計メモ

現段階では、開発効率と検証速度を優先して、`LeagueHome` に主要機能を集約した1ページ構成で実装を進めています。

これは最終UIではなく、以下を素早く確認するためのプロトタイプ用画面です。

- リーグ進行
- 成績更新
- 打線編集
- ローテ編集
- ブルペン編集
- 試合結果表示
- ニュース表示
- セーブ / ロード

最終的には、ゲームらしい導線にするため、機能ごとに画面を分割していく方針です。

現時点では、分割の第一歩として `LeagueHome` から `TeamManagement` と `LeagueInfo` へ遷移できる構成に着手しています。
今後は `LeagueHome` をホーム画面寄りに整理しつつ、編成や方針の編集は `TeamManagement`、順位表や成績確認は `LeagueInfo` 側へ段階的に移していきます。
現在の `LeagueHome` では、ホームとして必要な導線と進行情報を優先表示し、詳細な編成やリーグ情報は別画面で扱う方向へ移行中です。
ホームやリーグ情報のような確認画面は、縦に情報を積み上げるだけでなく、`1画面で全体像を掴めるダッシュボード型` を意識して、上部に要点を2列でまとめる方向へ整理していきます。
また、ホーム画面にはその日の予定や注目カードのような「進行前に見たい情報」を集めていく想定です。
加えて、前日の結果や最近の流れもホームで短く把握できる形を目指しています。
さらに、首位チームや主要タイトル争いがひと目で分かる簡易リーグ速報もホームに載せる方針です。
担当球団制の実装後は、ホーム内の予定や結果表示もまず担当球団の試合を優先して見せる方向で整理していきます。

### 想定している画面分割

1. タイトル画面
- はじめから
- つづきから
- 設定

2. ホーム画面
- 今日の状況
- 最新ニュース
- 次の試合 / 1日進行
- よく使う機能への導線

3. チーム管理画面
- チーム詳細
- 選手一覧
- 選手詳細
- 対右 / 対左打線
- 先発ローテ
- ブルペン
- チーム方針

現状は `LeagueHome` だけでなく `TeamManagement` からも `選手一覧` へ入れる導線を用意し、編成調整中に個別選手の情報へ飛びやすい形を優先しています。

4. リーグ情報画面
- 順位表
- 個人成績
- チーム一覧
- シーズン状況

## 開発方針

- まずは「シーズンが破綻なく回ること」を優先する
- その次に「数字を見て楽しいこと」を強化する
- その上で「ストリート野球らしい個性」を後から安全に足す
- 詳細リアル志向に進める場合も、段階的に粒度を上げる
- 早い段階で全部を再現しようとせず、拡張しやすい責務分離を保つ
